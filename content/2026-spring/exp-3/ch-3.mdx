## 全新篇章：自己动手写一个线程池！

在这一章，也就是 Chapter 3，我们会自己动手实现一个多线程调度器——线程池（ThreadPool）！听起来是不是超级厉害 (≧∇≦)ﾉ

这会有一点难度，但只要理清逻辑，耐心肯定能解决的。在写代码之前，我们需要先搞明白，**多线程到底会遇到什么绝境，以及前人们为了活命，被迫发明了什么工具**。

---

### 什么是“任务”和“队列”？

我们要建一个线程池，里面预先创建好几个**工作线程 (Worker Thread)**。既然要工作，首先得有任务。**主线程 (Main Thread)** 会源源不断地把任务提交进来，工作线程就从里面拿任务去执行。

这个存放任务的地方，就是**任务队列 (Task Queue)**。

在 C 语言中，一个“任务”到底是什么呢？
想象一下，这个线程池是一个通用的组件。它不关心具体的业务，今天可能被主线程叫去修图片，明天可能被叫去算工资。如果在线程池的代码里写死一句 `process_image();`，那这个组件就彻底绑死在图片处理上了，再也干不了别的活。

**我们怎么把“一段未知的代码”或者“一个动作”当成参数，塞进队列里传给工作线程呢？**
我们平时用的变量只能存数字、存字符，怎么才能存一个“动作”？

为了解决这个问题，C 语言的前辈们提供了一个极其强大的武器：**函数指针 (Function Pointer)**。

如果之前没有专门接触过底层语言，你可能会疑惑：**函数指针和普通指针有什么区别？**

- **普通指针**（比如 `int*` 或者 `char*`）：它们指向的是**数据**（变量、数组等）。它们告诉 CPU：“去这个内存地址拿点材料过来算一算”。
- **函数指针**：它指向的是**代码指令**。我们写好的函数代码，在编译运行后，其实也是静静地躺在内存里的某一块特定区域。函数指针保存的，就是这段代码在内存里的“入口首地址”。它告诉 CPU：“直接跳到这个地址，开始执行那里的动作”。

来看看 `thread_pool.h` 里的定义，虽然它的语法看起来<Spoiler>有点像天书</Spoiler>：

```c
typedef void (*thread_task_fn)(void*); // [!code highlight]

typedef struct {
  thread_task_fn fn;
  void* arg; // [!code highlight]
} ThreadTask;
```

- `typedef void (*thread_task_fn)(void*);`：怎么拆解这句语法呢？
  - 核心在中间的 `(*thread_task_fn)`，那个 `*` 明确告诉你，这是一个指针。
  - 右边的 `(void*)` 表示这个函数需要接收一个 `void*` 类型的参数。
  - 左边的 `void` 表示这个函数执行完不返回任何东西。
  - 连起来就是：定义了一种名为 `thread_task_fn` 的专门的“动作指针类型”。主线程只要把想要执行的函数地址存进 `fn` 里，工作线程从队列里拿出这个任务后，直接调用 `fn(arg);`，就能顺藤摸瓜，完美执行主线程临时指派的任何动作了。
- `void* arg;`：除了动作，还得有干活用的材料（参数）。既然工作线程一开始不知道具体是什么任务，自然也不知道参数到底是什么类型（是整型？是字符串？还是结构体？）。`void*` 就是 C 语言里的“万能盲盒”，任何类型的数据地址都能被塞进这个盲盒打包送进去。等真正进入了那个特定的任务函数内部，开发者自己心里清楚盒子里装的是什么，再把它强转回原来的真实类型拿出来用就行了。

为了高效利用内存，我们在 `ThreadPool` 结构体中把队列设计成了**环形队列 (Circular Queue)**：

```c
// 主线程塞入任务后，尾指针后移：
pool->queue_tail = (pool->queue_tail + 1) % pool->queue_capacity; // [!code highlight]

// 工作线程取出任务后，头指针后移：
pool->queue_head = (pool->queue_head + 1) % pool->queue_capacity; // [!code highlight]
```

用固定大小的数组配合取模运算 `%`，就能实现一个无限循环利用的流水线！<Spoiler>数学老师说过的取模终于有用了</Spoiler>

### 什么是“互斥锁”？

现在流水线有了，工作线程也准备就绪了。但如果你直接让他们去队列里拿任务，**灾难就会发生**。

想象一下：工作线程 A 看到 `queue_head` 是 0，准备去拿 0 号任务；但就在他刚伸出手的一瞬间，时间片轮转，工作线程 B 也看到了 `queue_head` 是 0，也去拿 0 号任务。最终不仅任务被重复执行，`queue_head` 还会被加两次，甚至引发段错误，整个队列瞬间崩溃。

这里的问题在于，读取 `queue_head`、取出任务、修改 `queue_head` 这一系列动作不是**原子**的（它们随时可以被打断）。
**是谁打断了它们？答案是：中断 (Interrupt)。**

如果你之前没有接触过底层，你可能会问：什么是中断？
其实，CPU 本质上是一个极其庞大但又无比刻板的**自动状态机**。它只要通了电，就会不知疲倦地一条接一条地执行指令。但如果它永远只执行眼前的代码，那系统一旦卡死或者进入死循环，整个电脑就彻底没救了。
所以硬件工程师发明了“中断”。这就像在 CPU 的大脑里埋了一个定时炸弹（时钟发生器）。每隔几毫秒，炸弹就会“砰”地炸一下（触发时钟中断）。不管 CPU 此时正在执行多么重要、多么神圣的代码，它都**必须无条件立刻停下手头的工作**，把控制权强制交还给操作系统的调度器。
正是依靠这种极其霸道的时钟中断机制，操作系统才能强行把 CPU 从一个线程手里抢过来，切给另一个线程，从而让你感觉几百个程序在“同时”运行。

**时序混乱**
如果工作线程 A 刚读完 `queue_head` 就遭遇了时钟中断，被强行“冻结”调度走；工作线程 B 刚好接管了 CPU，他就会读取到一样的 `queue_head`，然后拿走同一个任务。等 A 再被调度回来解冻时，他还以为自己手里拿到的是独一无二的任务。灾难就此发生！
像队列这种只要**时序不对**就会导致灾难、同一时刻只能被一个线程安全操作的资源，我们称之为**临界资源**。

为了解决这个绝境，前人们发明了一种机制来抵抗这种时序混乱。这种机制就像公共卫生间门上的那个“有人/无人”的标志牌——当你进去后，把牌子翻到“有人”（上锁），如果此时你被“中断”打断去休息了，别人因为拿不到锁，在外面就只能干等（空转或睡眠）；等你回来干完活出来时，再把牌子翻回“无人”（解锁）。

<Accordion title="CAS 与 Futex — 锁到底怎么造的？">

我们总是在说"用锁来保护"，那操作系统自己又是怎么造出锁的呢？

**第一层：关中断 + 自旋**

最原始、最粗暴的方法就是利用硬件特权：**关中断**（直接屏蔽 CPU 的时钟中断，谁也别想把我调度走！），然后加一个**自旋**（如果其他 CPU 核心上已经有别的线程拿到了锁，我就在 `while(1)` 里死等）。可以说：**互斥的最底层灵魂 ≈ 关中断 + 自旋**。

但我们在应用层不可能有"关中断"的特权（不然你写个死循环电脑就蓝屏了）。那应用层怎么办？

**第二层：硬件原子指令 CAS (Compare-And-Swap)**

CPU 硬件工程师专门设计了一条特殊的机器指令——CAS（x86 上叫 `CMPXCHG`）。它在**一条指令内**完成"比较并交换"：

```c
// 伪代码：CAS 的语义（硬件保证原子执行，不可被打断）
bool CAS(int *addr, int expected, int new_val) {
    if (*addr == expected) {
        *addr = new_val;
        return true;   // 交换成功，你抢到锁了！
    }
    return false;       // 别人已经改过了，你晚了一步
}
```

关键在于：这**整个操作是一条 CPU 指令**，硬件保证它不会被中断劈成两半。这就是原子性的真正来源——不是靠软件逻辑，而是靠硅片上的晶体管电路！

在多核 CPU 上，CAS 还需要**缓存一致性协议 (MESI)** 的配合。还记得第一章里讲过的 cache coherence 吗？当一个核心执行 CAS 修改了某个锁变量时，MESI 协议会立即使其他核心缓存中的旧值失效，确保所有核心看到的锁状态是一致的。

**第三层：Linux Futex (Fast Userspace Mutex)**

有了 CAS，我们就能在用户态尝试抢锁了。但如果抢不到，总不能一直自旋空转吧？这就是 Linux 的 Futex 大显身手的地方：

1. **快路径 (Fast Path)**：先在用户态用 CAS 试一下——如果锁没被人占，一条指令搞定，**根本不进内核**，开销几乎为零。
2. **慢路径 (Slow Path)**：如果 CAS 失败（锁被占了），才通过 `futex()` 系统调用陷入内核，让操作系统把当前线程挂起睡眠，放进等待队列。

这就是为什么 `pthread_mutex_lock` 在**没有竞争**的时候几乎免费（就一条 CAS 指令），只有在真的打架时才会付出系统调用的代价。绝大多数情况下，锁是没有竞争的——Futex 正是基于这个概率设计的，堪称工程智慧的极致体现。

</Accordions>

最后，我们来给这个伟大的发明下个彻底的定义。
这在计算机里被称为**互斥锁 (Mutex, Mutual Exclusion)**。
**什么是互斥？** 互斥就是人为地制造一种**绝对的排他性**。在微观的 CPU 状态机世界里，所有的指令都是交错执行、充满时序变数的。而互斥，就是硬生生地在这片混沌之中，强行劈出一块绝对安全的“静止时空”（临界区）。在这个时空里，你是唯一的霸主，没有中断的威胁，没有并发的干扰。

它本质上就是内存里的一个变量。但神奇的是，操作系统和 CPU 硬件为它提供了特殊的指令保证：**无论有多少个线程同时去抢这个锁，最终有且只有 1 个线程能成功把它设置为“已锁定”状态。**

```c
pthread_mutex_lock(&pool->mutex);   // 抢锁！抢不到的线程只能在门外乖乖排队睡觉（被操作系统挂起） // [!code highlight]
// ... 安全地操作队列 (取出任务，修改 queue_head) ...
// 这里的代码被称为“临界区”
pthread_mutex_unlock(&pool->mutex); // 完事了，把锁解开，操作系统会放下一个排队的线程进来 // [!code highlight]
```

只有在 `lock` 和 `unlock` 之间，代码才是绝对安全的。这也是互斥锁被发明的唯一目的：保护临界资源。

**来亲手感受一下 Data Race 有多可怕：**

下面这个小程序让两个线程各自给 `counter` 加 50000 次。把下面的代码保存为 `race.c`，用 `gcc race.c -o race -lpthread` 编译运行试试。按理说结果应该是 100000，但如果你不加锁，每次运行都会得到一个**不一样的随机数**：

```c
#include <pthread.h>
#include <stdio.h>

int counter = 0;
pthread_mutex_t lock;

void* worker(void* arg) {
    for (int i = 0; i < 50000; i++) {
        pthread_mutex_lock(&lock);   // [!code focus]
        counter++;
        pthread_mutex_unlock(&lock); // [!code focus]
    }
    return NULL;
}

int main() {
    pthread_t t1, t2;

    pthread_mutex_init(&lock, NULL); // [!code highlight]

    pthread_create(&t1, NULL, worker, NULL); // [!code highlight]
    pthread_create(&t2, NULL, worker, NULL); // [!code highlight]

    pthread_join(t1, NULL); // [!code highlight]
    pthread_join(t2, NULL); // [!code highlight]

    printf("counter = %d\n", counter); // 加锁：稳定 100000；不加锁：随机数！

    pthread_mutex_destroy(&lock); // [!code highlight]
    return 0;
}
```

试试把 `lock` 和 `unlock` 那两行注释掉再跑，你会亲眼看到 Lost Update 的威力。

| API                                   | 作用                                     |
| :------------------------------------ | :--------------------------------------- |
| `pthread_mutex_init(&lock, NULL)`     | 初始化一把锁（第二个参数用默认属性即可） |
| `pthread_mutex_lock(&lock)`           | 加锁，抢不到就阻塞等待                   |
| `pthread_mutex_unlock(&lock)`         | 解锁，让排队的线程进来                   |
| `pthread_mutex_destroy(&lock)`        | 销毁锁，释放资源                         |
| `pthread_create(&tid, NULL, fn, arg)` | 创建线程，让它执行 `fn(arg)`             |
| `pthread_join(tid, NULL)`             | 等待线程结束                             |

### 单锁不够怎么办？

现在工作线程很守规矩了，每次去拿任务都会先加锁。但这又引出了一个致命的逻辑死局：**如果当前队列是空的（主线程还没发任务），工作线程该怎么办？**

1. **死循环狂刷？**

   ```c
   while (queue_size == 0) { /* 疯狂加锁、解锁、查看 */ }
   ```

   绝对不行！这叫忙等待（Busy Waiting）。几个线程会在空队列前疯狂空转，让 CPU 占用率瞬间飙升到 100%，极度浪费资源。<Spoiler>风扇狂转的声音已经响起来了</Spoiler>

2. **睡一会儿再看？**
   ```c
   if (queue_size == 0) { sleep(1); }
   ```
   也不行。如果刚睡下 0.01 秒主线程就发任务了，系统就要白白干等 1 秒钟，响应极慢。<Spoiler>用户已经开始砸键盘了</Spoiler>

只靠一把互斥锁，我们陷入了死局。我们需要一种全新的机制：**当条件不满足（没有任务）时，让线程真正地挂起休眠（完全不占 CPU）；而当条件满足（新任务到来）时，其他线程能精准地把它唤醒。**

这就是操作系统为我们发明的神兵利器：**条件变量 (Condition Variable, `pthread_cond_t`)**。

你可以把条件变量理解为一个**等待队列**和一个**唤醒信号**。
当工作线程发现没任务时，就拿着互斥锁调用 `pthread_cond_wait` 挂起睡眠。当主线程往队列里塞了新任务后，就调用 `pthread_cond_signal` 或 `pthread_cond_broadcast` 发送信号，把挂起的工作线程唤醒起来干活。

**唤醒信号有两种，叫错人导致整个程序死锁怎么办？**
唤醒 API 有两种：`pthread_cond_signal` 只唤醒**一个**在等待的线程，而 `pthread_cond_broadcast` 唤醒**所有**等待的线程。
如果在复杂的逻辑里，不仅工作线程会因为没任务而挂起，主线程也会因为队列塞满而挂起（等待 `not_full` 条件）。
假设队列满了，主线程挂起了。此时两个工作线程 A 和 B 刚好干完活，队列空出了位置。工作线程 A 干完活后，如果只用 `signal`（随机唤醒一个线程），万一它唤醒的**不是主线程**，而是刚好也在挂起的**工作线程 B**，那主线程就永远挂死在那里了，整个程序彻底死锁！

<Callout title="核心观点 · broadcast" type="warn">

所以我们得出并发编程的一个重要观点：**如果不确定该唤醒谁，总是使用 `broadcast` 唤醒所有潜在可能被唤醒的线程！** 宁可错杀唤醒大家一起来抢锁，也不能漏掉那个真正该醒的线程。除了极度明确的单一场景，用 `broadcast` 永远是最安全的兜底方案。

</Callout>

<Accordion title="生产者 - 消费者模型 (Producer-Consumer Model)">

是不是觉得刚才的"主线程发任务，工作线程做任务"的模式很熟悉？
没错，这就是计算机科学中极其经典的**生产者-消费者模型**。

为什么要专门发明这种模型？因为它完美解决了**上下游处理速度不匹配**的绝境。
现实中，就像麦当劳的后厨：厨师（生产者）疯狂做汉堡，收银员（消费者）负责卖汉堡。如果厨师做好一个就必须亲自交到收银员手里，那收银员找零钱的时候厨师就只能干瞪眼等着。为了解决这个问题，人们发明了中间的"汉堡保温架"（队列）。厨师做完就扔到架子上继续做下一个，收银员有顾客就从架子上拿，两人彻底解耦，互不干扰，效率直接起飞！

在我们的线程池中：

- **生产者 (Producer)**：就是调用 `thread_pool_submit` 的主线程。它负责把任务源源不断塞进队列。如果队列满了，它就要在门外等 (`not_full`)。
- **消费者 (Consumer)**：就是那些 Worker 线程。它们在后台不断从队列取任务执行。如果队列空了，它们就要在休息室等 (`not_empty`)。

**这个伟大的模型在现实系统和内核中无处不在：**

- **管道 (Pipe)**：你在 Linux 命令行敲的 `ls | grep txt`，就是用管道（充当队列）把 `ls` 的输出（生产者）缓冲起来，作为 `grep` 的输入（消费者）。如果 `grep` 找得太慢，管道满了，`ls` 就必须阻塞等待；如果 `ls` 找得太慢，管道空了，`grep` 就必须挂起等待。
- **网卡收发机制**：网卡硬件芯片收到网络包（极速的生产者）后，为了不频繁触发中断打断 CPU，会把海量的网络包悄悄塞进内存的环形缓冲区（Ring Buffer），然后内核的网络协议栈线程（消费者）再去慢慢批量处理这些网络包。这也是典型的利用队列"削峰填谷"。
- **单生产者-单消费者 (SPSC)**：在某些极致性能场景下，如果我们能保证这个队列绝对只有 **1 个固定的生产者** 和 **1 个固定的消费者** 在操作，因为他们修改的队列指针（比如一个只改 head，一个只改 tail）天然互不干扰，高级程序员甚至能利用硬件的"内存屏障"实现出**无锁环形队列 (Lock-free Ring Buffer)**！连互斥锁开销都能完全省掉，性能极其炸裂！

**先来看一个最简单的"等通知"程序：**

把下面的代码保存为 `notify.c`，用 `gcc notify.c -o notify -lpthread` 编译运行。主线程准备好数据后唤醒工作线程，工作线程在收到信号之前一直挂起休眠：

```c
#include <pthread.h>
#include <stdio.h>

pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t  cond = PTHREAD_COND_INITIALIZER;
int ready = 0;

void* worker(void* arg) {
    pthread_mutex_lock(&lock);

    while (ready == 0) {                        // [!code focus]
        pthread_cond_wait(&cond, &lock);         // [!code focus]
    }

    printf("收到！开始干活\n");
    pthread_mutex_unlock(&lock);
    return NULL;
}

int main() {
    pthread_t tid;
    pthread_create(&tid, NULL, worker, NULL);

    // 模拟主线程准备数据...
    printf("主线程：准备数据中...\n");

    pthread_mutex_lock(&lock);
    ready = 1;                                   // [!code focus]
    pthread_cond_signal(&cond);                  // [!code focus]
    pthread_mutex_unlock(&lock);

    pthread_join(tid, NULL);

    pthread_mutex_destroy(&lock);
    pthread_cond_destroy(&cond);
    return 0;
}
```

注意几个关键细节：

1. 修改 `ready` 和调用 `signal` 都必须在**持有锁**的状态下进行。
2. 工作线程用的是 `while (ready == 0)` 而不是 `if`——具体原因马上在下一节揭晓。
3. `pthread_cond_wait` 会**自动释放锁再挂起**，被唤醒后又**自动重新加锁**——这一个调用干了三件事。

<Callout title="pthread 条件变量 API 速查" type="info">

| API                               | 作用                             |
| :-------------------------------- | :------------------------------- |
| `pthread_cond_init(&cond, NULL)`  | 初始化条件变量                   |
| `pthread_cond_wait(&cond, &lock)` | 释放锁 → 挂起 → 被唤醒后重新加锁 |
| `pthread_cond_signal(&cond)`      | 唤醒**一个**等待中的线程         |
| `pthread_cond_broadcast(&cond)`   | 唤醒**所有**等待中的线程         |
| `pthread_cond_destroy(&cond)`     | 销毁条件变量                     |

也可以用 `PTHREAD_MUTEX_INITIALIZER` / `PTHREAD_COND_INITIALIZER` 宏直接初始化全局变量，省掉 `init` 调用。

</Callout>

### wait 与 while

条件变量**必须**和互斥锁搭配使用！这是初学者最容易栽跟头的地方。
首先，我们要讲清楚 `pthread_cond_wait(&cond, &mutex)` 这个 API 到底在底层施了什么魔法。当你调用它时，它会在内核里**原子地**做两件事：

1. 把你手里的互斥锁 `mutex` 给解开（这样主线程才能拿到锁发任务）。
2. 把你这个线程塞进等待队列挂起睡眠。
   等到未来某一天，唤醒信号把你叫醒后，它又会在内部**自动去重新抢回**那把互斥锁。

看看正确的工作线程拿任务代码：

```c
pthread_mutex_lock(&pool->mutex); // 1. 先加锁

// 2. 必须用 while 循环判断条件！不能用 if！
while (pool->queue_size == 0 && pool->stop == 0) { // [!code focus]
    // 3. 挂起睡眠。
    // 注意：这句话会自动解开 mutex 锁！否则你拿着锁睡觉，主线程连发任务的锁都抢不到了！
    // 等它被唤醒时，又会在内部自动重新抢回 mutex 锁！
    pthread_cond_wait(&pool->not_empty, &pool->mutex); // [!code focus]
}

// 4. 被叫醒了，而且 queue_size > 0 啦！从队列中取出任务...

pthread_mutex_unlock(&pool->mutex); // 5. 关键：拿到任务后，第一时间先解锁！！！

// 6. 真正开始干活（千万别拿着锁干活，不然多线程就退化成串行了！）
task.fn(task.arg); // [!code focus]
```

**为什么醒来后还要用 `while` 循环再查一遍队列是不是空的？**

想象这个极其真实的场景：

1. 工作线程 A 在等待队列里睡眠。
2. 主线程放了一个新任务，发送了信号，唤醒了工作线程 A。
3. A 准备去抢互斥锁（`mutex`）。
4. **就在这短短的几毫秒内**，一个刚刚干完活、刚好路过的工作线程 B，顺手抢先一步拿到了锁，把那个新任务秒抢走执行了！
5. A 终于抢到了锁，走到队列一看：**空空如也！**

这就是所谓的**虚假唤醒 (Spurious Wakeup)**。你被叫醒了，但不代表轮到你拿锁的时候，条件依旧满足（任务还在）。
所以，A 醒来后的第一件事，必须用 `while` 循环再看一眼：队列还是空的吗？如果是，那就得乖乖回去接着睡。这就让 `while` 循环成为了逻辑上的必然，而不是死记硬背的规则。

<LabNote>

在线程池中, 你需要实现:

1. **`thread_pool_worker`**：实现工作线程的逻辑。安全地等待任务、取出任务，修改 `queue_size` / `queue_head` / `working_count`，并在干完活后发送信号更新状态。
2. **`thread_pool_submit`**：实现主线程提交任务的逻辑。安全地把新任务塞进环形队列尾部（更新 `queue_tail` 和 `queue_size`），并唤醒睡眠的工作线程。
3. **`thread_pool_wait`**：让主线程在这里乖乖等所有的任务都被处理完。
4. **`thread_pool_destroy`**：安全下班！通知所有线程退出，并用 `pthread_join` 回收它们。

</LabNote>

在这里，我们引出并发编程中关于条件变量的另一条**铁律（核心观点）**：
**总是在唤醒后，用 `while` 循环再次检查同步条件！** 千万别用 `if`。你必须假定这个世界充满了虚假唤醒。

<Accordion title="Mesa 语义 — while 循环的理论根基">

刚才我们从"别人会抢走任务"这个实际场景推导出了 `while` 的必要性。但这背后其实藏着一个更深层的理论故事。

1974 年 Tony Hoare 和 Per Brinch Hansen 设计了两套截然不同的语义：

**Hoare 语义**：`signal` 之后，发信号的线程**立即让出 CPU**，被唤醒的线程保证第一个拿到锁执行。这意味着醒来之后条件**一定成立**，用 `if` 就够了。理论上很优雅，但实现起来代价极高——每次 signal 都要做一次强制上下文切换。

**Mesa 语义**：`signal` 只是一个"建议"——它把等待线程从睡眠队列移到就绪队列，但**不会立即让出 CPU**。被唤醒的线程还得自己排队抢锁，等真正轮到它拿到锁的时候，条件可能早就被别人破坏了。

**现实中所有操作系统（POSIX、Windows、Java）都使用 Mesa 语义。** 原因很简单：Hoare 语义要求 signal 后立即切换上下文，在多核时代开销不可接受。

Mesa 语义甚至更极端：POSIX 标准**明确允许**实现层面的"虚假唤醒"——即使没有任何线程调用 `signal`，`pthread_cond_wait` 也可能自己醒过来！这是因为内核在处理信号投递 (signal delivery)、`fork()` 等内部事件时，可能会意外唤醒正在等待的线程。

所以结论是：`while` 循环不是"防御性编程"或者"以防万一"，而是 Mesa 语义下的**逻辑必然**。它和条件变量是一体的，缺了 `while` 的条件变量在理论上就是不完整的。

### 优雅谢幕

最后，主线程决定销毁线程池了（调用 `thread_pool_destroy`）。它不能直接强杀线程，必须妥善安置。

**1. 为什么需要 `working_count`？**
在 `thread_pool_wait` 里，我们需要等所有任务完成。很多同学觉得只要 `queue_size == 0`（队列空了）就算做完了。
大家思考一个场景：主线程放了最后一个任务，被线程 A 拿走了，此时 `queue_size` 变成 0。主线程一看队列空了，以为搞定了准备销毁。但它没看到的是，A 还在满头大汗地跑着最后一个任务的代码呢！
所以只有当**“待办列表是空的” (`queue_size == 0`) 且 “没有线程正在执行任务” (`working_count == 0`)** 时，才是真正的万事大吉。

**2. 怎么用 `pthread_join` 安全回收线程？**
主线程先加锁设置 `pool->stop = 1`，然后用 `pthread_cond_broadcast` 把所有因为队列为空而在睡眠的工作线程全部叫醒。
工作线程醒来一看 `stop == 1`，就知道没活干而且要退出了，直接 `return NULL;` 结束生命周期。

主线程必须确保所有的工作线程都已经真正退出。怎么确认？用 `pthread_join`！

`join` 这个概念在并发编程里极其常见，你可以把它生动地理解为现实生活中的**“死等”**：

- **NPY：等我洗个头就出门** ~~NPY 用了 `join(洗头)`，你只能在楼下干等~~
- **NPY：等我打完这局游戏就来** ~~NPY 用了 `join(游戏)`~~
- **舍友：等我修好这个 bug 就吃饭** ~~你只能饿着肚子 `join(修bug)`~~
- **导师：等我出差回来就讨论这个课题** ~~课题进度被 `join(出差)` 彻底阻塞了~~

所以，`pthread_join(thread, NULL)` 的含义就是：**等指定的线程彻底结束了，当前的代码才会继续往下走。**

```c
for (int i = 0; i < pool->thread_count; ++i) {
    pthread_join(pool->threads[i], NULL); // [!code focus]
}
```

`pthread_join` 会让主线程阻塞等在这里，直到对应的线程跑完了最后一行代码彻底退出，主线程才会走向下一个继续等。确认所有线程都安全退出后，正式释放内存，结束程序！

<Accordion title="扩展阅读：死锁 ">

学会了互斥锁和条件变量，你已经掌握了并发编程的两大武器。但在开始写代码之前，必须认识它们最可怕的副作用：**死锁 (Deadlock)**。

**经典场景：**

想象两个线程在操作两把锁：

```c
// 线程 A
pthread_mutex_lock(&lock_1);
// ... 做了一些事 ...
pthread_mutex_lock(&lock_2);       // 等待 lock_2，但 B 拿着它 // [!code highlight]

// 线程 B
pthread_mutex_lock(&lock_2);
// ... 做了一些事 ...
pthread_mutex_lock(&lock_1);       // 等待 lock_1，但 A 拿着它 // [!code highlight]
```

线程 A 拿着锁 1 等锁 2，线程 B 拿着锁 2 等锁 1。两个人互相等对方先松手，谁也不肯让步——**程序永远卡死**，没有任何报错，没有任何崩溃，就是纹丝不动。这就是死锁最恐怖的地方：**程序卡死，难以被察觉**。

<LabNote>

在 `pthread_pool` 的实验中, 不要忘记调用 `pthread_mutex_unlock` 释放锁, 避免造成死锁.

</LabNote>
**Coffman 四条件**

1965 年，Edward Coffman 证明了死锁发生必须**同时满足以下四个条件**：

1. **互斥 (Mutual Exclusion)**：资源同一时刻只能被一个线程持有
2. **持有并等待 (Hold and Wait)**：线程拿着一把锁的同时去等另一把锁
3. **不可抢占 (No Preemption)**：锁不能被强行夺走，只能由持有者主动释放
4. **循环等待 (Circular Wait)**：A 等 B、B 等 A，形成环路

破坏**任意一个条件**，死锁就不可能发生。

**所以我们有了这样的解决方法**：最常用的预防手段是破坏第 4 条"循环等待"：**规定所有线程必须按照同一顺序获取锁**。比如永远先拿 `lock_1` 再拿 `lock_2`，这样就不可能形成环路。Linux 内核甚至有一个叫 `lockdep` 的运行时检测工具，专门追踪所有锁的获取顺序，一旦发现潜在的顺序违规就立刻报警。

**哲学家就餐问题 (Dining Philosophers)**

这是 Dijkstra 在 1965 年提出的经典模型：5 位哲学家围坐圆桌，每两人之间放一把叉子（共 5 把）。吃饭需要同时拿起左右两把叉子。如果 5 个人同时拿起左手边的叉子，然后都在等右手边的——经典死锁！

解法之一：规定编号最小的哲学家必须**先拿右手边**的叉子。仅此一个人的策略反转，就足以打破循环等待，整桌人永远不会死锁。

在我们的线程池里虽然只有一把 `mutex`，不太可能出现教科书式死锁。但如果你忘了 `unlock`，或者在 `pthread_cond_wait` 之前没有正确持有锁，同样会导致程序永远卡住——调试时看起来和死锁一模一样。

</Accordion>
<Accordion title="扩展阅读：优化线程池">

写完这个线程池之后，你可能会好奇：真实世界里的线程池和并发系统是怎么工作的？

**线程数量怎么定？**

- **CPU 密集型任务**（纯计算，比如矩阵乘法）：线程数 ≈ CPU 核心数。开多了也没用，因为每个线程都在 100% 地使用 CPU，多出来的线程只会增加上下文切换的开销。
- **IO 密集型任务**（网络请求、磁盘读写）：可以开远多于核心数的线程。因为大部分线程在 `sleep` 等待 IO 返回，CPU 其实是空闲的。

**Work Stealing — 比共享队列更聪明的调度**

我们的线程池只有一个共享队列，所有线程都得抢同一把锁。当线程数和任务数暴增时，这把锁就成了瓶颈。现代高性能调度器（Go 语言的 Goroutine 调度器、Java 的 `ForkJoinPool`）使用了一种更巧妙的策略：**每个工作线程拥有自己的本地队列**。正常情况下，线程只从自己的队列里拿任务，完全不需要锁！只有当自己的队列空了，才会去**偷 (steal)** 别人队列尾部的任务。

这就是 **Work Stealing 算法**——把锁竞争从"所有人抢一把锁"降低到"偶尔两个人抢一下"，性能直接起飞。

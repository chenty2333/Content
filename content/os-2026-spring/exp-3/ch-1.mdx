## 为什么我们需要并行？

现代 CPU 通常都包含多个**核心**（Core），每一个核心在同一时刻只能执行一条**指令**（Instruction）。你之前写过的各种程序，在默认情况下都只运行在一个**线程**（Thread）中，而操作系统通常只会把这个线程**分配到一个 CPU 核心上**执行。这意味着，如果你什么都不做，即使你的电脑上有 8 个、16 个甚至更多的核心，事实上**只有一个核心**在执行你写的程序，其余的处于**空闲状态**。

~~这确实很令人难过，一方面是因为花钱买了电脑却发现只用了它一点点性能；另一方面，有些时候它真的跑的很慢！~~

比如你可能有一个有趣的功能，我们就暂时称它为 `do_something_good()` 吧，你可能真的会想对 1000 个输入进行你有趣的操作？所以出于直觉你一下子就写出了你最擅长的 `for` 循环（相信大部分人应该更喜欢写 `for` 吧），熟练的填上大一就写烂的 `i++`，编译执行！

```c lineNumbers
int sum = 0;

// [!code word:1000]
for (int i = 0; i < 1000; i++) { // [!code highlight]
    sum += do_something_good(i);
}
```

然而大部分时候它看起来也是在瞬间完成的，<Spoiler>就像你在大一的时候写的 C 语言作业一样快</Spoiler>。

毕竟你写的 `do_something_good()` 应该也没有多复杂就是了。你的电脑还是没有那么笨的，但是**并不代表它真的一直都很聪明**。总有一天你发明出了一些比较复杂的函数，比如一个<Spoiler>自动</Spoiler>完成你作业的函数 `do_my_homework()`，用在操作系统每周的作业上，<Spoiler>学习通有那么多题</Spoiler>，写一题要 10 秒的话，写 50 题就需要快九分钟！

```c lineNumbers
for (int i = 0; i < 50; i++) {
    do_my_homework(); // 10 秒一次 // [!code highlight]
}
```

如果你还记得一些计算机组成原理的知识的话，<Spoiler>或许你已经忘了</Spoiler>，你应该知道 CPU 的一个核心一个时候只能做一件事情。~~哪怕忘了也没有关系~~，希望你对做实验的那台**上古电脑和仪器**有点印象，从 `PC` **程序计数器**出来的东西一次只有一个，实际上它确实只有一个，所以单核心的 CPU 和人脑一样，它**一次也只能做一道题**。

你买电脑的时候肯定看过吧，至少知道电脑有几个核心，或者起码**知道它肯定不是单核的**。反正你突然觉得如果可以让我的电脑的好几个核心**同时帮我写作业**应该是一件很酷的事情，<Spoiler>但是还是不要去破解学习通哈</Spoiler>。

<Callout title="概念复习 · 并行" type="info">
  把任务分配给**多个核心同时进行处理**，这个过程就叫**并行 (Parallelism)**。
  并行的核心思想，就是把原本必须**顺序执行**的工作拆分成**多个可以同时进行**的部分，让它们在不同的
  CPU 核心上**并发**执行。
</Callout>

<Callout title="并发和并行的区别" type="warn">

并发（Concurrency）关注的是**程序的结构设计**，指的是一个程序能够**同时处理**多个任务，这些任务的完整**周期在时间上重叠**，但**不一定**真的在同一瞬间完成。简单得多，并发是**同时管理很多事情**。

并行（Parallelism）关注的是程序的实际执行，指的是多个任务真的在**同一时刻进行**，是真正意义上的**同时做很多事情**，需要硬件的支持才能真正实现，而**多核心 CPU** 就是这个硬件。

</Callout>

<Callout title="概念复习 · CPU 核心与线程" type="info">

CPU 核心（Core）是硬件层面的**物理执行单元**。每个核心在**同一时刻**只能高效执行一条指令流（单指令流）。多核
CPU 就像有多条**并行的流水线**。

线程（Thread）是**软件层面的执行单元**，代表一段**独立的指令序列**（可以运行的代码）。操作系统负责把**线程调度**到可用的 CPU 核心上去执行。

</Callout>

## 从手动并行到 OpenMP

我们当然可以**手动创建和管理**多个线程（例如使用 `std::thread` 或 `pthread`）。这种方式虽然非常灵活，但需要自己处理线程创建、任务划分、数据同步等问题，**代码复杂容易出错**。我们不希望自己分配，<Spoiler>学会高效偷懒是进步的开始</Spoiler>，创建线程，四五个可能还好，那几十个呢？

<Callout title="延伸：代码生成（Codegen）" type="idea">

面对以上这类的问题，你可能会说，既然写几十个也是**一种机械的任务**，为什么不能让计算机**自己做**呢？

当然，这是当今现代软件开发里非常常见的一种手段，名字叫做**代码生成**（Codegen），即使用一份代码**生成**更多的其他结构性的代码。比如，你的一个程序中，可能有很多地方是**基本重复**的，但是每一份之中都有几个值的名字不一样，这种时候**提取函数不一定**能解决问题，如果可以写一个**生成绝大多数重复代码**的字符串程序，那或许就可以解决。

当然，最适合生成程序的地方毋庸置疑，是**编译器**；你应该已经见过这样的概念了，比如 C 语言的 `#define`，它是一种**编译器指令**，用于替换代码的文本本身，一般称为**宏**（Macro）。

</Callout>

所幸，OpenMP 可以为我们创建这些**处理线程的程序**，它也是一套**编译器指令**（Compiler Directives），通过在代码里加入 `#pragma omp` 开头的指令，编译器会自动完成，**创建数量合适的线程**、**循环迭代分配**并**处理线程同步**。

最基础的用法是在 `for` 循环前添加 `parallel for` 指令：

```c
#pragma omp parallel for // [!code highlight]
for (int i = 0; i < 1000; i++) {
    // 一些比较慢的操作
    // 比如做 1000 次 printf()
}
```

此时，编译器会在生成的程序中为你创建并管理线程，把这 1000 次任务**均匀**分给它们；操作系统就会**调度这些线程**到 CPU 核心上去执行。

## 为什么需要 Reduction ？

看起来很完美对吧？但如果我们直接把刚才求和的代码加上这句魔法：

```c
int sum = 0;

#pragma omp parallel for
for (int i = 0; i < 1000; i++) {
    // [!code word:sum]
    sum += calculate_complex_math(i); // [!code focus]
}
```

**程序会算出一个完全错误的随机数！为什么？**

我们虽然只写了一句 `sum += calculate_complex_math()`，但是在硬件执行时，分为了**Load + Add + Store**三步。

让我们复习一下计组课上的知识<Spoiler>希望你还没忘光</Spoiler>：计算机中所有的**数据和指令**都存放在**内存**中，CPU 硬件电路里有一个 **PC（程序计数器）** 指针，指向当前要执行的指令在内存中的位置。每执行完一条指令，PC 就自动 +1，指向下一条。变量 `sum` 的值也在内存中，**Load** 指令会把它从内存加载到 CPU 的**寄存器**中，**加法器**执行 **Add** 指令将两个寄存器中的数相加，最后 **Store** 指令把结果写回内存保存。

当两个线程**同时**对 `sum` 进行"Load → Add → Store"时，混乱就发生了。假设下面的指令按时间顺序依次发生：

| 时间 | 线程 1                | 线程 2                | 内存中 sum 的值 |
| ---- | --------------------- | --------------------- | --------------- |
| t1   | **Load** sum (读到 0) |                       | 0               |
| t2   |                       | **Load** sum (读到 0) | 0               |
| t3   |                       | Add 5 → **Store** sum | **5**           |
| t4   | Add 3 → **Store** sum |                       | **3** ← 覆盖！  |

线程 2 辛辛苦苦算出来的 5 被线程 1 的 Store **直接覆盖**了，因为线程 1 在 t1 时刻读到的还是旧值 0，它根本不知道线程 2 已经更新过了。这就是经典的 **Lost Update（丢失更新）** 问题。

<Accordions type="multiple">
  <Accordion title="延伸 · 程序是一台状态机">

你有没有想过，程序到底是**怎么"自动运行"起来**的？

本质上，一个正在运行的程序就是一台**状态机（State Machine）**。它的核心可以用一个简洁的数学公式描述：

$$S_{n+1} = F(S_n,\ I_n)$$

状态转移函数 $F$ 有两个参数 $S_n$ 和 $I_n$。其中 $S_n$ 是计算机在第 $n$ 步的**完整状态**——包括 PC 的值、所有寄存器的内容、以及整个内存的数据；$I_n$ 是这一步接收到的**外部输入**（比如键盘输入、外部信号）；$F$ 就是"执行一条指令"这个动作本身。每一个时钟周期，CPU 做的事情就是：读取当前状态 $S_n$ 和输入 $I_n$ → 执行 $F$ → 输出新状态 $S_{n+1}$ → 覆盖。如此循环往复，程序就"跑起来了"。

**"程序"并不是什么神秘的东西，它就是一个确定性的状态转移函数。** 给定相同的初始状态和输入序列，结果永远一样。

而 Data Race 之所以危险，**恰恰是因为多线程打破了这个确定性**——两个线程各自基于自己看到的"旧快照" $S_n$ 去计算 $S_{n+1}$，然后互相覆盖了对方的输出。

  </Accordion>
  <Accordion title="延伸 · 从状态机到冯诺依曼架构">

状态机告诉我们程序**是**什么，而**冯诺依曼架构**告诉我们它在真实硬件上**怎么跑**。

上面的状态转移函数 $F$，在硬件上是由五大组件协作完成的：

1. **控制器（Control Unit）**：读取 PC 指向的指令，解码，决定"这一步 $F$ 要做什么操作"。
2. **运算器（ALU）**：执行具体的计算——加减乘除、逻辑运算，对应 $F$ 中的实际计算部分。
3. **存储器（Memory）**：同时存放**指令和数据**——这是冯诺依曼架构最核心的特征。
4. **输入设备 / 输出设备（I/O）**：对应公式中的 $I_n$，负责与外部世界交互。

重点来了：既然指令和数据**住在同一块内存里，共享同一条总线进出**，那么 CPU 想取下一条指令（Fetch）的同时，还想 Load/Store 当前的数据，但只有**一条路**，只能排队——这就是著名的**冯诺依曼瓶颈**。

这也从另一个角度解释了为什么 `sum += ...` 必须被拆成 Load、Add、Store 三步：因为总线一次只能做一件事，取数据和写数据没法在同一瞬间完成。

解决冯诺依曼瓶颈的关键思路是**加缓存**。内存虽大但慢，CPU 寄存器虽快但少，所以现代 CPU 在两者之间插入了多级缓存：**L1 → L2 → L3**，越靠近核心越快、容量也越小。其中 L1 离核心最近、速度最快（通常只需 1-3 个时钟周期），但也最小（通常只有几十 KB）。关键是：**每个核心都有自己独立的 L1 和 L2 缓存**，只有 L3 是所有核心共享的。

在 L1 层面，现代 CPU 采用了**哈佛架构**的设计——把**指令缓存（I-Cache）和数据缓存（D-Cache）分开**，各走各的路，取指令和读写数据可以同时进行，互不干扰。这算是在冯诺依曼架构的大框架下，用缓存层面的哈佛架构来"打补丁"。

  </Accordion>
</Accordions>

**怎么解决？**

先退一步想想：`sum += x` 到底是什么操作？它把**一堆值逐步折叠成一个最终结果**——1000 个计算结果，经过 1000 次 `+`，最终变成一个数。这种**多变一**的操作，在计算机科学里有一个专门的名字：**归约（Reduction）**。

Reduction 的特征很简洁：

- 有一个**二元运算符**：`+`、`*`、`max`（取最大值）、`min`（取最小值） 等等
- 有一个**初始值（单位元）**：加法是 `0`，乘法是 `1`，`max` 是负无穷...
- 不断把新值和累积结果**合并**，最终从 $n$ 个值变成 $1$ 个值

**初始值是由计算的性质决定的**。对于加法：任何数加上初始值 `0`，都保持原值。对于乘法，任何数乘以 `1` 也保持原值。`max` 和 `min` 同理。

注意，**并不是所有的并行循环都需要 reduction**。如果你的循环里每次迭代只写自己的数据，比如 `result[i] = f(i)`，或者只是重复 `printf()`，各个线程互不干扰，一个简单的 `parallel for` 就够了。只有当多个迭代需要**往同一个共享变量上累积**时——也就是做 reduction 操作时——才需要额外处理。

理解了这一点，解法就自然了：既然 reduction 最终要把所有值合并，那就让**每个线程先在自己的局部变量上做局部归约**，等所有线程都算完了，再把各自的局部结果按运算符**合并成最终值**。

OpenMP 把这个过程封装成了一个指令：**`reduction`**。

```c
#pragma omp parallel for reduction(+:sum) // [!code focus]
for (int i = 0; i < 1000; i++) {
    sum += calculate_complex_math(i);
}
```

`reduction(+:sum)` 告诉编译器三件事：

1. 为每个线程创建一个 `sum` 的**局部副本**。
2. 因为运算符是 `+`，所以局部副本的初始值设为 `0`（加法的单位元。如果是 `*`，初始值就是 `1`）。
3. 循环结束后，自动把所有局部副本按 `+` 合并到最终的 `sum` 变量里。

## 为什么还需要调度？

Reduction 解决了数据正确性的问题。但性能呢？

如果这 1000 次 `calculate_complex_math(i)` 的计算量**不均匀**——有的是极其复杂的运算，有的几乎瞬间完成——那么 OpenMP 默认的 **`static`（静态分配）** 就会出问题：它把 1000 次迭代均分成 4 份，每个线程 250 次。如果线程 A 分到的全是重计算，其他三个线程早就算完了，整个程序的耗时完全**被最慢的那个线程拖垮**。

**怎么解决？**

改变分配策略：**`dynamic`（动态分配）**。不提前分配了，而是维护一个公共的任务队列。谁先做完手上的迭代，谁就去队列里再领一个。能者多劳。

```c
#pragma omp parallel for schedule(dynamic) reduction(+:sum) // [!code focus]
for (int i = 0; i < 1000; i++) {
    sum += calculate_complex_math(i);
}
```

注意这里 `schedule(dynamic)` 和 `reduction(+:sum)` **可以组合使用**——前者管任务怎么分配，后者管共享变量怎么合并，各司其职。

**隐藏的性能瓶颈！**
这个公共任务队列本身就是一个**临界资源**<Spoiler>就像寝室里只有一个卫生间</Spoiler>，每次只允许一个线程来领任务。如果每做完一次迭代就跑去排队领下一个，排队的开销甚至可能比计算本身还大！

所以我们让每个线程一次领走**一块（chunk）** 迭代：

```c
#pragma omp parallel for schedule(dynamic, 8) reduction(+:sum) // [!code focus]
for (int i = 0; i < 1000; i++) {
    sum += calculate_complex_math(i);
}
```

这里的 `8` 就是 chunk 大小：每次领 8 个迭代回去算，减少排队次数。

除了 `schedule(dynamic)` 我们还有另一种选择，`schedule(guided)`。

`guided` 结合了静态和动态的优点：一开始任务多的时候，一次分一大块（比如 100 次迭代），极大减少排队开销；随着剩余任务减少，每次分的块越来越小（最后可能只分 1 个），防止有线程在最后领了一大堆重计算而其他线程全在等它。
也就是：**前期大块分减少排队开销，后期小块分保证负载均衡。**

<Accordions type="multiple">
  <Accordion title="扩展知识：缓存一致性 (Cache Coherence)">

为了让 CPU 跑得更快，每个核心都有自己专属的超高速**缓存 (Cache)**。"每个核心有自己的缓存"这件事本身就埋下了新的隐患：如果两个核心的 L1 缓存里各自存了同一个变量的**不同版本**，谁说了算？当核心 A 修改了自己缓存里的变量 `sum` 时，核心 B 缓存里的 `sum` 还是旧数据。如果没有同步机制，核心 B 就会用错数据。

同样道理，CPU 还有一种 per-core 的缓存叫 **TLB**（用来加速虚拟地址到物理地址的转换），也面临多核之间的同步问题。

为了保持数据一致，一旦某个核心修改了共享变量，CPU 硬件在底层就必须通过总线疯狂发送信号（比如通过 **MESI 协议**），去作废或更新其他所有核心里的旧缓存行。这个过程叫做**缓存一致性（Cache Coherence）** 维护。

这也就是为什么在写多线程程序时，**尽量让不同的线程去操作不同的数据，减少对同一块内存的频繁竞争**，是提升多核性能的终极奥义——这正是 `reduction` 的思路：每个线程先写自己的局部变量，最后才合并，避免了运行过程中的缓存同步风暴。

  </Accordion>
  <Accordion title="扩展知识：TLB 刷新 (TLB Shootdown)">

还记得前面提到的 **TLB** 吗？它是每个核心用来缓存"虚拟地址 → 物理地址"映射的硬件组件。和 L1 缓存一样，每个核心有自己独立的 TLB，而且因为 L1 分成了 I-Cache 和 D-Cache，TLB 也对应地分成了 **ITLB**（指令侧）和 **DTLB**（数据侧）。

程序代码里用的都是"虚拟地址"，CPU 每次读写内存都要把虚拟地址转换成"物理地址"，这个转换需要查**页表**。TLB 就是页表的高速缓存，避免每次都去内存里查表。

现在想象这样一个场景：操作系统决定把某块内存页回收了，或者把它换到了硬盘上。OS 必须立刻去修改页表，宣告这个虚拟地址失效了。但这个时候，其他 3 个核心的 TLB 里可能还缓存着旧的页表映射！如果它们继续用旧的映射去写内存，就会发生灾难性的越界错误。

为了解决这个致命问题，OS 会触发一种叫 **TLB Shootdown（TLB 击落）** 的操作：当一个核心修改了页表，OS 会立刻向其他所有核心发送一种特殊的硬件中断（**IPI，核间中断**）。其他核心收到信号后，必须立刻停下手中的工作，清空自己 TLB（包括 ITLB 和 DTLB）里的旧缓存，然后再恢复运行。

这个过程极其昂贵，因为它强行打断了其他核心的流水线！所以在多核时代，操作系统设计者们也是绞尽脑汁，想尽一切办法来减少这种跨核心的同步开销。

  </Accordion>
</Accordions>
